RUNBOOK · MARVIS-EVAL

Mavis-Eval 复现手册

三种姿势把 benchmark 跑起来: 人工手动操作、指挥 AI agent 操作、完整复现 17 个 task 的结果。读完这一页, 任何拿到 repo 的人 30 分钟内可以跑出第一个 report。

v0.1 面向: 工程师 + 评估者 + 面试官 配套: demo.html (设计文档) Author: Koutian Wu; GitHub: ktwu01

00为什么有这份手册

Demo (demo.html) 讲的是 设计: 评估维度、schema、关卡 (gate)、判分器 (judge)。这份 RUNBOOK 讲的是 怎么把它跑起来。

A 路径 · 人工手动

你在终端里, 一条条敲 mavis-eval 命令。适合: 第一次用、想搞清楚每一步在做什么、调试某个 case。

B 路径 · 指挥 agent

把这份手册丢给 Claude Code / Codex / Cursor, 让它代你跑。适合: 批量评估、自动化集成、CI 流水线。

C 路径 · 完整复现

从空 repo 到 17 份 report 全跑完。适合: 拿到 repo 想验证 demo 里的数字真不真。

核心约定。 一次 evaluate 等于: 一个 case JSON + 一个 run 目录 喂给 mavis-eval evaluate, 输出一份 report JSON。case 描述任务、run 是 agent 的实际产出 (trajectory + 文件), report 是评估结论。

01先决条件 (Prerequisites)

1.1 必装

  • Python 3.11+ (python3 --version 验证)
  • Git, 用于 clone repo
  • 一个能写文件的工作目录

1.2 可选 (想用真实 Mavis 跑 task 才需要)

  • 本地装好 MiniMax 桌面 app (/Applications/MiniMax.app)。
  • 跑 task 前: 先打开桌面 app。Electron 一启动就会 spawn 内置 daemon, CLI 才能连。不要直接跑 mavis start; v3.0.27 上它会因 better-sqlite3 打包问题崩, 见 MAVIS_CLI_SOP.md §5。
  • 验证 daemon 健康: ~/.mavis/bin/mavis status 应返回 {"status":"running","port":15321,...}; cat ~/.mavis/daemon.pid 应看到 "owner":"electron" (这是"桌面 app 启的"标记)。
  • (进一步) 能跑 judge 模型的 API key (OpenAI / Anthropic / 等等), 用来跑 L3 LLM judge。注意: 当前 17 份 report 是在 judge 不可用的情况下产出的, 仅靠 L1 / L2 deterministic 关卡, 故事依然成立, 见 demo 里"实测"。
关于 Mavis CLI。 公开 MiniMax 文档目前提到的是 mmx-cli (在 npm 上 npm install -g mmx-cli), 不是我们这里用的 mavis CLI。mavis 是桌面 app 在安装时附带的本地 daemon-management CLI, 实际入口在 /Applications/MiniMax.app/Contents/Resources/resources/daemon/cli.js, 经 ~/.mavis/bin/mavis 软链。本仓库通过这条非公开但稳定的路径成功驱动了 17 个 case 的 end-to-end 评估。完整调用约定见 MAVIS_CLI_SOP.md。

1.3 一次性安装

# clone repo
git clone <repo-url> MARVIS-EVAL
cd MARVIS-EVAL

# 没有外部依赖, 纯 Python 标准库就能跑 deterministic 评估
# 想跑 judge 的话另外装一下你选的 LLM SDK, 例如 openai 或 anthropic

# 验证安装
python3 -m mavis_eval --help
# 应该看到 7 个子命令: validate / list-cases / evaluate / judge-payload / report / compare / run-mavis
为什么没有 pip install。 这是 MVP, 故意保持零外部依赖, 你 clone 下来就能跑。pyproject.toml 里 dependencies 是空的, 不是漏写, 是故意的。

02A 路径: 人工手动跑 推荐第一次用

目标: 在终端里, 把一个具体 case (例如 travel_plan_001) 评估出一份 report。所有命令都是真实的, 我们的 17 份 report 就是这么跑出来的。

1

验证 case 文件合法

先确认 case 文件结构 OK, schema 没坏。

python3 -m mavis_eval validate cases/examples
# 期望输出: validated N cases from M files
2

看一下都有哪些 case

python3 -m mavis_eval list-cases cases/examples
# 输出 17 行, 每行一个 case_id + 标题
3

挑一个 case 看它的内容

例如东京家庭游 (L4 高难度):

python3 -c "import json; cases = json.load(open('cases/examples/smoke_seed_cases.json')); c = [x for x in cases if x['case_id'] == 'travel_plan_001'][0]; print(json.dumps(c, indent=2, ensure_ascii=False))" | less

看里面的 user_instruction, input_modalities, success_criteria, rubric。这就是 agent 要解的题。

4

看一下 run 目录里 agent 的产出

每个 case 对应 runs/<case_id>/ 一个目录, 里面是 agent 实际跑出来的东西。

ls runs/travel_plan_001/
# 输出大致:
# output/        agent 写出的文件 (这里是 itinerary.md)
# trajectory.jsonl  agent 每一步调用了什么工具、看到了什么
# state/         agent 操作前后的环境快照

cat runs/travel_plan_001/output/itinerary.md | head -30
5

跑 deterministic 评估

这一步把 L1 (gating) + L2 (quality) 的关卡评出来。不需要 LLM, 几十毫秒就好。

python3 -m mavis_eval evaluate cases/examples/smoke_seed_cases.json \
  --case-id travel_plan_001 \
  --run-dir runs/travel_plan_001 \
  --deterministic-only \
  --out reports/travel_plan_001.json
6

读 report

python3 -c "import json; r = json.load(open('reports/travel_plan_001.json')); print(f'pass: {r[\"pass\"]}  gating: {r[\"gating_pass\"]}  judge_available: {r[\"judge\"][\"available\"]}')"
# pass: True  gating: True  judge_available: False

L1+L2 全过, L3 judge 没接, 当前判为 PASS。详细看下面 读懂 report。

7

看一个失败案例

把命令换成 excel_anomaly_001, 重跑第 5、6 步:

python3 -m mavis_eval evaluate cases/examples/smoke_seed_cases.json \
  --case-id excel_anomaly_001 \
  --run-dir runs/excel_anomaly_001 \
  --deterministic-only \
  --out reports/excel_anomaly_001.json

python3 -c "import json; r = json.load(open('reports/excel_anomaly_001.json')); print(json.dumps(r['assertions']['failed_gates'], indent=2))"

会看到 agent 的 CSV 列名错了 (q1_sales 而不是 previous_quarter_sales), L1 schema 关卡当场抓住, FAIL。这就是 benchmark 的价值: 表面看 agent "完成了任务", 但格式不合规, 下游会炸。

第一次跑完了。 你已经从 case 出发, 看到 run 产出, 跑了评估, 看到一个 pass 一个 fail。整个 pipeline 就这 7 步, 后面只是把它批量化。

03B 路径: 指挥 agent 跑 推荐批量评估

目标: 把这份手册丢给一个 AI coding agent (Claude Code / Codex / Cursor / 其他), 它代你跑。用于 CI、定时回归、人不在的时候。

3.1 给 agent 的 prompt 模板

下面这一段, 直接复制到 agent 的对话框, 替换 <...> 占位符即可。

你是一个 evaluation 工程师。在 MARVIS-EVAL 项目根目录下工作。

任务:
1. 跑 `python3 -m mavis_eval list-cases cases/examples`, 获取所有 case_id。
2. 对每一个 case_id, 依次跑:
   python3 -m mavis_eval evaluate cases/examples/smoke_seed_cases.json \
     --case-id <case_id> \
     --run-dir runs/<case_id> \
     --deterministic-only \
     --out reports/<case_id>.json
3. 跑完后, 聚合:
   python3 -m mavis_eval report reports --out reports/summary.json
4. 读 reports/summary.json, 报告: 总数 / pass 数 / fail 数 / 每个 fail 的根因 (从 assertions.failed_gates 里抽 message 字段)。
5. 如果有 fail, 不要试图修复 case 或 run, 只汇报。

约束:
- 不要修改 cases/ 或 runs/ 下任何文件
- 不要 commit, 不要 push
- 全部命令在项目根目录跑
- 200 字以内汇报结果

3.2 给 agent 的"接真实 Mavis"prompt (进阶)

如果你想让 agent 不光评估, 还**自己驱动 Mavis 把 case 跑一遍**, 用 run-mavis 子命令:

前置 (必须按顺序):
  1. 打开 /Applications/MiniMax.app (让 Electron 启动内置 daemon)。
     不要跑 `mavis start`; v3.0.27 上它会因 better-sqlite3 打包问题崩。
  2. 验证 daemon 健康:
       ~/.mavis/bin/mavis status         # 期望: running, port 15321
       cat ~/.mavis/daemon.pid           # 期望: "owner":"electron"
  3. 如果上面不健康, 停下来读 docs/MAVIS_CLI_SOP.md, 不要硬冲。

对每个 case_id, 跑:
  python3 -m mavis_eval run-mavis cases/examples/smoke_seed_cases.json \
    --case-id <case_id> \
    --fixtures-root fixtures \
    --timeout-s 300

这个命令会:
  - 把 fixtures/<case_id>/ 下的输入文件拷进 runs/<case_id>/
  - 调 mavis session new, 启动一次新会话
  - 把 case 的 user_instruction 投到 mavis
  - 等 mavis 跑完, 把 trajectory 转成 trajectory.jsonl
  - 跑 deterministic 评估, 写 reports/<case_id>.json

如果 mavis daemon 报 better-sqlite3 错, 先停下来读 docs/MAVIS_CLI_SOP.md, 别硬扛。

3.3 给 agent 的"加判分器 (judge)"prompt (进阶)

前置: 已配置 OPENAI_API_KEY (或你选的 LLM provider 的 key)。

对每个 case_id, 两步:

第 1 步: 生成 judge payload (给判分器看的证据包)
  python3 -m mavis_eval judge-payload cases/examples/smoke_seed_cases.json \
    --case-id <case_id> \
    --run-dir runs/<case_id> \
    --out judge_payloads/<case_id>.json

第 2 步: 把 prompts/deliverable_judge_v1.0.txt 作为 SYSTEM, 把 payload 作为 USER, 调 LLM,
        把返回的 JSON 存到 judge_results/<case_id>.json

第 3 步: 把 judge result 合到最终 report
  python3 -m mavis_eval evaluate cases/examples/smoke_seed_cases.json \
    --case-id <case_id> \
    --run-dir runs/<case_id> \
    --judge-result judge_results/<case_id>.json \
    --out reports/<case_id>.json

注意:
- judge 模型不要看 trajectory 里的 agent self-reasoning, 只看 tool calls / observations / final artifact
- judge 输出必须是 strict JSON, 看 prompt 里 line 33 后面的 schema
给 agent 的安全围栏。 在 agent prompt 里明确写 "不要修改 cases/ 和 runs/"。否则 agent 看到 fail 会"贴心地"去改 case 让它通过, 这是评估反模式 (Goodhart's law), 必须堵住。

05读懂 report

每份 reports/<case_id>.json 大致长这样:

{
  "case_id": "travel_plan_001",
  "title": "Plan a constrained Tokyo family trip",
  "scenario": "planning_and_decision_support",
  "difficulty": "L4",

  // === 三个 gate 的总判定 ===
  "gating_pass": true,         // L1: deterministic gating (file_exists, no_forbidden_tool, ...)
  "rubric_pass": false,        // L3: judge rubric, false 因为 judge 没跑
  "pass": true,                // final = gating_pass && (rubric_pass || judge_unavailable_fallback)

  // === 详细 assertion 结果 ===
  "assertions": {
    "gating_assertions": [...],  // L1 关卡
    "quality_assertions": [...], // L2 关卡
    "failed_gates": [],         // 失败的 assertion 列表
    "gating_pass": true,
  },

  // === judge 部分, 没跑时长这样 ===
  "judge": {
    "available": false,
    "requires_human_audit": true   // 标记需要人审, 这是 escalation 机制
  },

  // === 元信息 ===
  "evaluated_at": "2025-05-15T...",
  "schema_version": "1.0"
}

5.1 关键字段速查

字段含义
gating_passL1 关卡 (硬性约束: 文件在不在、禁用工具有没有用、URL 合不合法)。FAIL 直接淘汰。
rubric_passL3 judge 给的语义评分通不通过。judge 没跑时为 false。
pass最终判定。gating fail 一定 fail, gating pass 后视 rubric / quality 决定。
assertions.failed_gates所有失败 assertion 的明细, 含 expected / observed / message, 这是 debug 起点。
requires_human_auditjudge 自评不确定时翻为 true。这是 LLM judge → 人审的 escalation 通道。
safety_redline是否踩了安全红线 (例如调了禁用 API、发了禁止域名 URL)。true 直接 fail。

06常见问题 (Troubleshooting)

Q: mavis-eval validate 报 schema error

case 文件缺字段。看 schemas/case.schema.json 第 200 行 around required, 对照 case 文件补齐。Schema 是 single source of truth。

Q: run-mavis 报 Cannot find module 'better-sqlite3'

说明 daemon 不是由桌面 app 启动的 (大概率你跑了 mavis start)。v3.0.27 的 daemon 必须由桌面 app 的 Electron 子进程启动, 才能加载到 bundled 的 native SQLite 模块。解法: 关掉所有 mavis 进程, open /Applications/MiniMax.app, 等 2 秒, 再 ~/.mavis/bin/mavis status 验证 (期望 "owner":"electron")。完整说明见 MAVIS_CLI_SOP.md §5。

Q: report 里 pass=true 但 rubric_pass=false, 矛盾吗

不矛盾。judge (L3) 没跑时, 有一个 "deterministic-only fallback" 规则: gating 全过 + quality 全过 → 整体判 pass, 但标 requires_human_audit=true。这是 demo 第 06 章讲的三道关卡里的 fallback 策略。

Q: 我改了一个 case, report 数字变了, 怎么 review

跑 python3 -m mavis_eval compare --prior reports.bak --new reports, 它会给你 case 级 diff + 整体 Wilson CI + McNemar 显著性。不要肉眼比 17 个 JSON 文件。

Q: judge 用什么模型最合适

当前 prompt 是 model-agnostic。GPT-4 / Claude Sonnet / Gemini 2.5 都能跑。Demo 第 08 章里讲了为什么 judge 不能信 agent 的 self-reasoning, 只能看 trajectory + final artifact。如果你接的模型不支持 strict JSON output, 加一个 retry 包装层。

Q: 我想加一个新 case

三步: (1) 在 cases/examples/smoke_seed_cases.json 里加一个 case object, 按 schema; (2) 在 fixtures/<new_case_id>/ 放输入文件; (3) 在 runs/<new_case_id>/ 准备 agent 产出 (或用 run-mavis 让 Mavis 实际跑一次)。然后跑 validate + evaluate。

07一页速查 (Cheat Sheet)

# === 评估单个 case (最常用) ===
python3 -m mavis_eval evaluate cases/examples/smoke_seed_cases.json \
  --case-id <CASE_ID> \
  --run-dir runs/<CASE_ID> \
  --deterministic-only \
  --out reports/<CASE_ID>.json

# === 批量跑全部 17 个 case ===
python3 scripts/run_all_cases.py

# === 聚合 ===
python3 -m mavis_eval report reports --out reports/summary.json

# === 跨版本对比 ===
python3 -m mavis_eval compare --prior reports/v1 --new reports/v2 --out reports/diff.json

# === 真实跑 Mavis (先开桌面 app, 然后再跑) ===
open /Applications/MiniMax.app                # 让 Electron 启动 daemon
~/.mavis/bin/mavis status                     # 期望 running, port 15321
python3 -m mavis_eval run-mavis cases/examples/smoke_seed_cases.json \
  --case-id <CASE_ID> --fixtures-root fixtures --timeout-s 300

# === 接 LLM judge ===
python3 -m mavis_eval judge-payload ... --out judge_payloads/X.json
# <调 LLM, 把返回的 JSON 存到 judge_results/X.json>
python3 -m mavis_eval evaluate ... --judge-result judge_results/X.json --out reports/X.json

# === 验证 case schema ===
python3 -m mavis_eval validate cases/examples

# === 列出所有 case ===
python3 -m mavis_eval list-cases cases/examples
